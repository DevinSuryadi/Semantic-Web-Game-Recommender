KONTEKS PROJECT GAMEFEEL KG

Judul:
GameFeel KG: Sistem Rekomendasi Game Berdasarkan Kemiripan Semantik Menggunakan RDF, Ontology, dan SPARQL.

Tujuan project:
Membangun sistem rekomendasi game berbasis Semantic Web yang membandingkan game berdasarkan kemiripan "feel", bukan hanya genre, rating, popularitas, atau platform. Feel game direpresentasikan sebagai gabungan atribut semantik seperti mood, theme, gameplay mechanic, game mode, subgenre, combat style, perspective, difficulty, pacing, platform, art style, tag, dan quality tier.

Contoh:
Jika pengguna menyukai Elden Ring, sistem tidak hanya mencari game bergenre Action RPG atau Open World, tetapi mencari game lain yang memiliki karakteristik semantik seperti dark fantasy, challenging combat, exploration, boss fight, mysterious world, high difficulty, dan atmospheric experience.

Prinsip penting:
- Project harus tetap fokus pada Semantic Web: RDF, Ontology, SPARQL, Knowledge Graph, dan explainable recommendation.
- Tidak menggunakan Streamlit.
- Tidak menggunakan machine learning kompleks.
- Frontend menggunakan React + TypeScript + Vite.
- Backend menggunakan Node.js + TypeScript + Express.
- Package manager menggunakan pnpm.
- Prisma + SQLite hanya digunakan sebagai staging/local database.
- Prisma tidak menggantikan RDF, ontology, atau Apache Jena Fuseki.
- Apache Jena Fuseki tetap menjadi SPARQL endpoint utama untuk rekomendasi.
- Data utama untuk rekomendasi dibaca dari RDF/Knowledge Graph melalui SPARQL.

REKOMENDASI BRANCHING

Jangan memakai nama branch data-scraping untuk tahap awal ini, karena project mengambil data melalui RAWG API, bukan scraping halaman web. Istilah yang lebih tepat adalah ingestion atau pipeline.

Branch yang direkomendasikan untuk tahap data awal:
feature/rawg-data-ingestion

Alasan:
- "rawg" menjelaskan sumber data.
- "data-ingestion" menjelaskan proses mengambil dan menyimpan data awal.
- Tidak menyiratkan web scraping.
- Cocok untuk pekerjaan: setup RAWG API client, fetch game, simpan ke Prisma, dan ekspor raw dataset.

Alternatif nama branch:
- feature/data-awal-rawg
- feature/rawg-data-pipeline
- feature/pilot-dataset

Urutan branch yang disarankan:
1. feature/project-setup
   Untuk setup monorepo, pnpm workspace, Express API, React frontend, Prisma, env, dan struktur folder.

2. feature/rawg-data-ingestion
   Untuk mengambil data objektif dari RAWG API dan menyimpannya ke staging database.

3. feature/semantic-curation
   Untuk menambahkan label semantik manual/terkurasi seperti mood, theme, mechanic, pacing, dan difficulty.

4. feature/ontology-design
   Untuk membuat ontology Turtle/OWL, class, object property, datatype property, domain, dan range.

5. feature/rdf-generation
   Untuk membuat script TypeScript yang mengubah dataset curated menjadi RDF/Turtle.

6. feature/fuseki-sparql
   Untuk setup Fuseki, upload ontology/RDF, membuat query search, detail, dan recommendation.

7. feature/api-sparql-integration
   Untuk menghubungkan Express API ke endpoint SPARQL Fuseki.

8. feature/web-recommender-ui
   Untuk membuat frontend React yang menampilkan search, detail, score, dan alasan rekomendasi.

Jika ingin mulai dari tahap data awal sekarang, branch terbaik adalah:
feature/rawg-data-ingestion


TIMELINE TUGAS

1. Diskusi Ide Awal
Deskripsi:
Rapat kelompok menentukan ide proyek "GameFeel KG" sebagai sistem rekomendasi game berbasis Semantic Web dan membagi tugas awal.

Output yang harus ada:
- Judul project.
- Rumusan masalah.
- Tujuan project.
- Daftar anggota dan pembagian tugas.
- Gambaran alur sistem.


2. Pencarian Referensi
Deskripsi:
Mencari referensi terkait Semantic Web, RDF, ontology, Knowledge Graph, SPARQL, Apache Jena Fuseki, dan dokumentasi RAWG API.

Output yang harus ada:
- Ringkasan referensi Semantic Web.
- Ringkasan konsep RDF triple.
- Ringkasan ontology dan OWL.
- Ringkasan SPARQL.
- Catatan endpoint dan field data RAWG API.


3. Pengumpulan Data Dasar
Deskripsi:
Mengambil data objektif game menggunakan RAWG API. Data objektif mencakup id, title, slug, released, genre, platform, developer, publisher, rating, metacritic, playtime, tags, description, dan background_image.

Output yang harus ada:
- Script RAWG API client.
- Data raw dari RAWG, disarankan disimpan sebagai data/raw/rawg_games.json.
- Data staging di Prisma SQLite.
- Minimal 10 game untuk pilot, lebih baik 30-50 game jika waktu memungkinkan.


4. Kurasi Atribut Semantik
Deskripsi:
Melakukan pelabelan manual untuk menambahkan data subjektif/semantik seperti mood, theme, gameplay mechanic, pacing, difficulty, art style, combat style, perspective, subgenre, game mode, dan quality tier.

Output yang harus ada:
- Kamus semantik final.
- Dataset curated, disarankan data/curated/gamefeel_dataset.csv.
- Kolom sumber/evidence mapping jika memungkinkan.
- Aturan bahwa label semantik harus berasal dari kamus, sedangkan istilah lain masuk sebagai Tag.


5. Finalisasi Dataset
Deskripsi:
Membersihkan data hasil RAWG API dan pelabelan manual, lalu menyatukannya ke dalam satu dataset final.

Output yang harus ada:
- Dataset final dalam CSV atau JSON curated.
- Data mentah tetap disimpan terpisah dari data curated.
- Format kolom konsisten.
- Tidak ada nilai semantik bebas di luar kamus, kecuali Tag.

Saran struktur dataset:
- data/raw/rawg_games.json
- data/curated/gamefeel_dataset.csv
- data/README.md untuk menjelaskan sumber data dan aturan kurasi.


6. Desain Ontologi
Deskripsi:
Mendefinisikan class seperti Game, Genre, Mood, Theme, GameplayMechanic, GameMode, SubGenre, Platform, Developer, Publisher, dan property seperti hasMood, hasGenre, hasTheme, availableOn, developedBy, publishedBy.

Output yang harus ada:
- ontology/gamefeel_ontology.ttl
- Class ontology.
- Object properties.
- Datatype properties.
- Domain dan range.
- Validasi ontology dengan Protege.



7. Pembuatan Script TypeScript
Deskripsi:
Menulis script menggunakan TypeScript untuk memetakan dataset final menjadi RDF triple. Di screenshot sebelumnya tertulis Python, tetapi untuk project ini harus TypeScript agar konsisten dengan context dan stack.

Output yang harus ada:
- Script generate RDF.
- Mapping dari kolom dataset ke RDF properties.
- Escape literal RDF dengan benar.
- Output Turtle valid.


Nama script yang disarankan:
- apps/api/src/scripts/generateRdf.ts
atau
- scripts/generate-rdf.ts


8. Konversi ke RDF
Deskripsi:
Mengeksekusi script konversi dan memvalidasi hasil file Turtle.

Output yang harus ada:
- rdf/gamefeel_data.ttl atau rdf/gamefeel_10_games.ttl untuk pilot.
- RDF dapat dibuka/di-load ke Fuseki.
- Semua resource semantik menjadi individual/instance, bukan string bebas.



9. Setup Apache Jena Fuseki
Deskripsi:
Mengunduh, menginstal, dan mengatur konfigurasi server lokal Apache Jena Fuseki.

Output yang harus ada:
- Fuseki berjalan di http://localhost:3030.
- Dataset bernama GameFeel.
- Endpoint query: http://localhost:3030/GameFeel/query.
- Endpoint update/upload jika diperlukan: http://localhost:3030/GameFeel/update.



10. Deployment Dataset
Deskripsi:
Mengunggah file RDF/Turtle dan ontology ke dataset GameFeel di Fuseki.

Output yang harus ada:
- ontology/gamefeel_ontology.ttl di-load ke Fuseki.
- rdf/gamefeel_data.ttl di-load ke Fuseki.
- Dataset dapat di-query dari Fuseki UI.


11. Perancangan Kueri SPARQL
Deskripsi:
Menyusun query SPARQL untuk mencari game, melihat detail game, dan menghitung skor rekomendasi berdasarkan intersection atribut semantik.

Output yang harus ada:
- sparql/search_game.rq
- sparql/game_detail.rq
- sparql/recommend_by_game.rq
- Query recommendation menggunakan bobot similarity.


12. Testing SPARQL
Deskripsi:
Menguji langsung query SPARQL di Fuseki UI. Hasil rekomendasi dan alasannya harus muncul.

Output yang harus ada:
- Screenshot hasil query search.
- Screenshot hasil query detail.
- Screenshot hasil query recommendation.
- Minimal satu contoh input, misalnya Elden Ring.


13. Penyusunan Proposal
Deskripsi:
Merekap logbook pengerjaan, menjelaskan rancangan sistem, dan menyusun proposal/laporan.

Output yang harus ada:
- Latar belakang.
- Rumusan masalah.
- Tujuan.
- Batasan masalah.

ATURAN PEMETAAN RAWG KE SEMANTIK

Data objektif:
- RAWG id -> rawgId
- name -> title
- slug -> slug
- released -> releaseDate
- genres -> Genre dan kemungkinan SubGenre
- platforms -> Platform
- developers -> Developer
- publishers -> Publisher
- rating -> rating
- metacritic -> metacriticScore
- playtime -> playtime
- description -> description
- background_image -> imageUrl
- tags -> sumber utama mapping semantik

Prioritas mapping:
1. Tag RAWG
2. Description
3. Genre inference
4. Manual curation

Catatan:
Untuk kualitas dataset tugas, manual curation tetap boleh dipakai, terutama karena atribut seperti mood, theme, dan feel bersifat subjektif dan tidak selalu tersedia langsung dari RAWG.

METODE REKOMENDASI

Sistem mencari kesamaan atribut semantik antar game menggunakan RDF dan SPARQL. Similarity score dihitung berdasarkan intersection resource semantik yang sama.

Bobot similarity:
- Genre sama: +3
- SubGenre sama: +3
- Mood sama: +2
- Theme sama: +2
- Gameplay mechanic sama: +2
- GameMode sama: +2
- CombatStyle sama: +2
- Difficulty sama: +1
- Pacing sama: +1
- Perspective sama: +1
- ArtStyle sama: +1
- Platform sama: +1
- Tag sama: +1
- QualityTier sama: +1

Output rekomendasi harus explainable:
- slug game rekomendasi
- title game rekomendasi
- similarity score
- alasan rekomendasi, misalnya Genre: RPG, Mood: Dark, Theme: Dark Fantasy, Mechanic: Boss Fight.

CATATAN PERBAIKAN DARI RANCANGAN SCREENSHOT

1. "Data scraping" kurang tepat.
Gunakan "RAWG API ingestion" atau "pengumpulan data melalui RAWG API".

2. "Script Python" harus diganti.
Project ini menggunakan TypeScript, jadi tugasnya adalah "Pembuatan Script TypeScript".

3. CSV perlu diputuskan secara eksplisit.
Jika timeline menyebut CSV, maka buat dataset curated CSV agar laporan konsisten. Kalau tidak ingin CSV, ubah timeline menjadi JSON/TypeScript dataset. Untuk tugas akademik, CSV curated lebih mudah dijelaskan.

4. Protege sebaiknya disebut sebagai alat validasi ontology.
Jika ontology ditulis manual, gunakan kalimat "divalidasi menggunakan Protege", bukan "dibuat menggunakan Protege".

5. Branch main jangan dipakai untuk eksperimen.
Gunakan feature branch untuk setiap tahap agar riwayat kerja lebih rapi.
